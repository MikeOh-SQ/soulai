# 전체 프로세스

이 문서는 ADHDQQ.COM의 실제 구현 기준으로, 사용자의 입력이 어떤 JSON 값으로 저장되고 어떤 계산과 AI 보조를 거쳐 화면 결과가 만들어지는지 정리합니다.

## 1. 시작과 ID

사용자는 새 ID를 만들거나 기존 ID의 최신 기록을 불러옵니다. 예시 placeholder는 `test01`입니다.

저장 기본 구조:

```json
{
  "id": "test01",
  "createdAt": "ISO",
  "updatedAt": "ISO",
  "fileName": "test01-YYYYMMDD-HHMMSS.json",
  "currentStep": "id",
  "tests": {
    "asrs": [],
    "dsm5": [],
    "game": {}
  },
  "report": null,
  "plan": {
    "suggestions": [],
    "chat": []
  },
  "dtx": {
    "stage": "stage1",
    "totalScore": 0,
    "scores": {
      "plangame": 0,
      "test1": 0,
      "test2": 0
    }
  },
  "tutorials": {
    "plan": false,
    "build": false,
    "chop": false
  },
  "planGame": {
    "stage": "stage1",
    "score": 0,
    "goals": []
  }
}
```

내부 키는 과거 호환성을 위해 `asrs`, `dsm5`를 유지하지만, 사용자 화면에는 `자가보고 선별`, `증상 기준 체크`로 표시합니다.

`dtx`, `tutorials`, `planGame`는 메인 검사만 사용할 때는 없을 수 있고, `DTx`/`plangame` 실험 경로를 탔을 때 추가됩니다.

## 2. SELF: 자가보고 선별

질문 소스는 [config/asrs.json](/home/sqmini/soulai/config/asrs.json)입니다.

### 입력 방식

- 총 6문항
- 0~4 척도
- 문항별 예시 제공
- 문항별 이미지 `public/asrs01.png` ~ `public/asrs06.png` 표시

### 저장 요약

```json
"asrs": {
  "responses": [3, 4, 2, 3, 4, 3],
  "attention_score": 12,
  "attention_max": 16,
  "hyperactivity_score": 7,
  "hyperactivity_max": 8,
  "total_score": 19,
  "total_max": 24,
  "positive_count": 6
}
```

### 로컬 계산

- 1~3번: 2점 이상이면 유의미 응답
- 4~6번: 3점 이상이면 유의미 응답
- `positive_count >= 4`면 추가 평가를 고려할 수 있는 선별 신호로 설명
- `attention_score`, `hyperactivity_score`를 나눠 리포트에 사용

### AI 사용

자가보고 선별 직후 `/api/ai/asrs-analysis`를 호출할 수 있습니다. 화면에는 일반 명칭으로 표시하지만 서버 내부 함수명은 기존 호환성을 위해 `generateAsrsAnalysis()`를 유지합니다.

AI 출력:

```json
{
  "summary": "string",
  "attention": "string",
  "hyperactivity": "string",
  "guidance": "string"
}
```

## 3. SYM: 증상 기준 체크

질문 소스는 [config/dsm-5.json](/home/sqmini/soulai/config/dsm-5.json)입니다.

### 입력 방식

- 총 23문항
- Yes/No 응답
- 1~9번: 부주의
- 10~18번: 과잉행동/충동성
- 19~23번: 추가 확인
- 문항별 이미지 `/dsmimages/d1.png` ~ `/dsmimages/d23.png` 표시

### 저장 요약

```json
"dsm5": {
  "responses": [true, false, true],
  "inattention_true_count": 6,
  "hyperactivity_true_count": 4,
  "contextual_true_count": 3,
  "total_true_count": 10,
  "subtype": "부주의형 가능성"
}
```

### 로컬 계산

- 부주의 6개 이상, 과잉행동/충동성 6개 미만: `부주의형 가능성`
- 과잉행동/충동성 6개 이상, 부주의 6개 미만: `과잉행동·충동형 가능성`
- 두 영역 모두 6개 이상: `복합형 가능성`
- 두 영역 모두 6개 미만: `무증상 범위`
- 추가 확인 문항은 일상 부담과 맥락 해석을 보강합니다.

## 4. REACT: 반응성 테스트

반응성 테스트는 `signal_detection -> go_nogo -> balance_hold` 순서로 진행됩니다. 사용자는 화면 버튼으로 특정 테스트만 골라 건너뛸 수 없습니다.

### 테스트 1. 신호 찾기

- 연습 6회, 본 시행 60회
- 목표 자극에만 반응
- 자극 시간 500ms
- ISI 1000/1250/1500ms
- 100ms 미만 반응은 `anticipatory`
- 진행 중 안내 문구는 최소화

저장 핵심값:

```json
{
  "test_type": "signal_detection",
  "omission_rate": 0.13,
  "mean_reaction_time": 702,
  "reaction_time_variability": 142,
  "tau": 214,
  "late_phase_drop": 0.12,
  "anticipatory_count": 1
}
```

### 테스트 2. 멈춤 버튼

- 연습 8회, 본 시행 50회
- `o.gif`는 누르고 `x.gif`는 누르지 않음
- GIF 사양: 1024x1024, 25프레임, 0.50초, loop count 1
- 자극 시간 500ms
- ISI 800/1000/1200ms
- 성공 기준: Go 자극 시작 후 220ms 이후부터 자극 종료 전까지
- Apple touch device는 시각/터치 지연을 고려해 380ms 보정
- No-Go는 최소 2회 이상 Go 뒤에 배치

저장 핵심값:

```json
{
  "test_type": "go_nogo",
  "go_count": 38,
  "nogo_count": 12,
  "go_hit_count": 33,
  "commission_errors": 3,
  "commission_rate": 0.25,
  "fast_error_rate": 0.08,
  "mean_go_reaction_time": 421,
  "premature_response_count": 2
}
```

### 테스트 3. 균형 유지

- 본 검사 30초
- 모바일 센서 가능 시 `sensor`
- iPhone/iPad에서 센서 권한 실패 또는 센서 미감지 시 `long_touch`
- PC에서는 센서가 있어 보이더라도 모바일/터치 조건이 아니면 센서 모드로 들어가지 않음
- 롱터치 모드에서는 손가락 좌표를 실제로 반영하고, 터치가 떨어지면 가운데 점은 숨김

저장 핵심값:

```json
{
  "test_type": "balance_hold",
  "stable_duration_pct": 74.6,
  "spike_count": 3,
  "total_movement": 91.2,
  "input_source": "long_touch"
}
```

## 5. RESULT: 통합 리포트

리포트는 `/api/ai/insights` 요청으로 생성되지만, 최종 report 본문은 서버의 결정론적 계산을 우선합니다.

### 서버 계산

[server.js](/home/sqmini/soulai/server.js)의 `computeAssessmentMetrics(record)`가 다음을 계산합니다.

- 자가보고 부주의/충동성 점수
- 증상 기준 체크의 부주의/과잉행동·충동성/추가 확인 개수
- 목표 놓침 비율
- 잘못된 반응 비율
- 반응시간 변동성
- Tau
- 후반부 저하
- 성급 반응
- 안정 유지 시간
- 주관-객관 일치 여부
- 일상 부담 수준

### 화면 구성

- Hero summary
- 주관적 보고 vs 객관적 반응
- 부주의 탭
- 충동성 탭
- 일상 부담
- 계획으로 이동 CTA

## 6. PLAN: 실행 계획

`plan`은 두 단계로 생성됩니다.

1. 서버 fallback: `buildPlanForMetrics(metrics)`가 유형별 기본 계획 3개를 만듭니다.
2. Gemini 보조: `plan.suggestions`, `openingMessage`, 채팅 답변을 더 자연스럽게 조정합니다.

분기:

- `inattention`: 과제 분할, 시각적 단서, 환경 통제
- `impulsivity`: 지연 행동, 심호흡, 자기 점검
- `combined`: 부주의 보완 + 충동 조절 혼합
- `very_low`: 현재 상태 유지 루틴

## 7. 한 줄 요약

- `SELF`: 자가보고 선별, 로컬 점수 + AI 짧은 해석
- `SYM`: 증상 기준 체크, 로컬 유형 집계 + AI 짧은 해석
- `REACT`: 브라우저 기반 수행 데이터 수집
- `RESULT`: 저장 수치 기반 deterministic 리포트
- `PLAN`: 서버 유형별 계획 + Gemini 조정 채팅

## 8. 운영/보조 흐름

최근 구현 기준으로 메인 검사 플로우 바깥에 아래 보조 경로가 추가되어 있습니다.

### shortcut 진입

- `/?route=asrs`, `/?route=report&id=test01` 같은 쿼리로 특정 단계에 바로 진입할 수 있습니다.
- `/?shortcut=reactivity&test=signal_detection&id=test01`은 반응성 overview로,
  `/?shortcut=reactivity-result&test=go_nogo&id=test01`은 완료 화면으로 바로 여는 디버그/데모용 진입입니다.
- `report` 또는 `plan` shortcut으로 열었고 기록이 있으면 `ensureInsights()`를 즉시 실행해 최신 리포트를 맞춥니다.

### JSON 보기와 admin 모달

- 메인 앱 상단에는 진행 중 record를 바로 확인하는 `JSON 보기` 모달이 있습니다.
- admin 모달에서는 `database/*.json`의 메타 목록을 읽어 기존 기록을 다시 불러올 수 있습니다.
- 이 기능은 문서화/디버깅/수동 검증용 운영 보조 장치입니다.

### DTx 숲 허브와 플랜 게임

- `DTx` 경로는 `/dtx?id=test01`입니다.
- 이 화면은 메인 검사 결과 이후의 보조 루프로, `record.dtx.scores`를 합산해 숲 stage를 계산합니다.
- `plangame`은 `record.plan.suggestions`를 기본 목표로 가져와 3개의 목표 카드로 펼치고,
  목표 1회 완료마다 `+10점`, 재완료 쿨다운 `60분`을 적용합니다.
- 누적 점수는 `stage1`~`stage5` 배경으로 시각화됩니다.

### 구조도 map

- `/map`은 JSON 파일, 프런트 로컬 스키마, 서버 metrics, AI 프롬프트 입력, 최종 화면 연결을 시각화한 정적 설명 화면입니다.
- 실서비스 단계가 아니라 문서/설계 확인용 보조 도구입니다.
