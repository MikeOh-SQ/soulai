# 반응성 테스트 3종 사양서

## 문서 목적

이 문서는 ADHD 선별·감별 보조 웹앱 안에서 사용할 반응성 테스트 3종의 기능 요구사항을 정의한다.
본 문서는 Codex가 실제 구현에 바로 활용할 수 있도록, 각 테스트의 목적, 규칙, 입력/출력 데이터, 점수 계산 방식, 상태 전이, 실패 조건, 저장 형식까지 포함한다.

## 공통 원칙

* 모든 테스트는 웹앱에서 실행 가능해야 한다.
* 한 테스트의 총 길이는 30초~90초 이내를 기본으로 한다.
* 사용자 피로를 줄이기 위해 조작 방식은 최대한 단순해야 한다.
* 모든 테스트는 검사 모드와 훈련 모드로 나뉜다.
* 검사 모드는 측정 일관성이 중요하므로 난이도를 고정한다.
* 훈련 모드는 적응형 난이도를 허용한다.
* 결과는 사용자용 자연어 문장과 시스템용 JSON으로 동시에 저장 가능해야 한다.
* 최종 진단이 아니라 선별·감별 보조용 데이터라는 점을 항상 유지한다.

## 전체 테스트 구성

1. 부주의 측정형 테스트: 신호 찾기
2. 충동성 측정형 테스트: 멈춤 버튼 (Go/No-Go)
3. 과잉행동/자기조절 측정형 테스트: 균형 유지

# 1. 부주의 측정형 테스트: 신호 찾기

## 목적

사용자의 지속주의, 선택적 주의, 반응시간 안정성을 측정한다.

## 핵심 측정 지표

* omission_errors: 눌러야 할 목표 자극을 놓친 횟수
* hit_count: 목표 자극에 올바르게 반응한 횟수
* false_alarm_count: 누르면 안 되는 자극을 잘못 누른 횟수
* mean_reaction_time: 정답 반응 평균 시간(ms)
* reaction_time_variability: 반응시간 표준편차(ms)
* tau: 유난히 늦게 반응한 구간의 크기
* late_response_count: 지나치게 늦은 반응 횟수
* late_phase_drop: 전반부 대비 후반부 성능 저하 정도
* anticipatory_count: 100ms 미만 반응 수

## 기본 규칙

* 화면 중앙에 자극이 1개씩 제시된다.
* 사용자는 특정 목표 자극이 나오면 탭 또는 클릭한다.
* 목표가 아닌 자극에는 반응하지 않는다.
* 자극은 가변 간격으로 반복 제시된다.
* 자극이 사라진 뒤에도 다음 자극이 시작되기 전까지는 같은 시행의 입력으로 인정된다.
* 이 구간은 쉬는 시간이 아니라 입력 가능 시간으로 UI에서 따로 표시한다.

## 예시 자극 구조

* 목표 자극: 파란 별
* 비목표 자극: 빨간 원, 초록 세모, 노란 네모 등
* 실제 구현에서는 색상 이상 사용자 접근성을 위해 모양 차이도 반드시 포함한다.

## 검사 모드 상세 사양

* 총 trial 수: 60
* 목표 자극 비율: 25%
* 자극 표시 시간: 500ms
* 자극 간 간격(ISI): 1000ms / 1250ms / 1500ms 랜덤
* 반응 허용 시간: 자극 시작 후 다음 자극 시작 전까지
* 연습 trial: 6
* 연습 단계에서 규칙 재안내 제공
* 검사 시작 전 3초 카운트다운 표시

## 훈련 모드 상세 사양

* 초기 trial 수: 20
* 사용자가 연속 성공 시 자극 속도 증가 가능
* 목표 자극 판별 난이도 조정 가능
* 3회 이상 누락 발생 시 속도를 낮추고 시각 피드백 강화

## 점수 계산 예시

* omission_rate = omission_errors / target_count
* reaction_time_variability = 정답 반응의 표준편차
* tau = 가장 느린 상위 10% 반응 평균 - 전체 평균 반응시간
* late_phase_drop = 전반부 대비 후반부 정확도 하락 비율
* 최종 부주의 해석은 점수보다 `omission_rate`, `reaction_time_variability`, `tau`, `late_phase_drop`를 우선 사용

## 해석 원칙

* omission_errors가 높고 reaction_time_variability가 크면 지속주의 취약 가능성이 높다.
* 전반부 대비 후반부에서 누락과 지연 반응이 늘어나면 주의 유지 저하 가능성을 시사한다.
* false_alarm이 높으면 순수 부주의 외 충동성 영향도 같이 의심 가능하다.
* 100ms 미만 반응은 예측 반응으로 보고 정상 반응 계산에서 제외한다.

## UI 요구사항

* 한 화면에 자극 1개만 표시
* 현재 진행도 표시 바 제공
* 검사 모드에서는 점수 실시간 공개 금지
* 검사 모드에서는 정답/오답 오버레이를 표시하지 않음
* 자극이 사라진 뒤에도 입력 가능 시간이 남아 있음을 시각적으로 표시

## 저장 JSON 예시

```json
{
  "test_type": "signal_detection",
  "mode": "assessment",
  "target_count": 10,
  "non_target_count": 30,
  "hit_count": 8,
  "omission_errors": 2,
  "false_alarm_count": 1,
  "omission_rate": 0.2,
  "mean_reaction_time": 742,
  "reaction_time_variability": 165,
  "tau": 214,
  "late_response_count": 3,
  "late_phase_drop": 0.18,
  "anticipatory_count": 2,
  "interpretation": "누락 오류와 반응시간 변동성이 비교적 높아 지속주의 취약 가능성을 시사합니다."
}
```

# 2. 충동성 측정형 테스트: 멈춤 버튼 (Go/No-Go)

## 목적

사용자의 반응 억제, 성급한 반응 경향, 충동적 오반응 패턴을 측정한다.

## 핵심 측정 지표

* commission_errors: 누르면 안 되는 자극에 반응한 횟수
* go_hit_count: 눌러야 하는 자극에 올바르게 반응한 횟수
* go_omission_count: 눌러야 하는 자극을 놓친 횟수
* mean_go_reaction_time: Go trial 평균 반응시간
* commission_rate: No-Go 실패 비율
* fast_error_rate: 200ms 미만 반응 비율
* anticipatory_count: 100ms 미만 반응 수
* level_summary.pattern: 충동형 / 혼합형 / 비교적 균형 패턴

## 기본 규칙

* 대부분의 자극은 눌러야 한다.
* 특정 금지 자극이 나오면 절대 누르면 안 된다.
* 빠르게 반응하는 것보다 규칙을 지키는 것이 더 중요하다.
* 자극이 사라진 뒤 짧은 입력 가능 시간이 이어지며, 이것도 같은 시행 반응으로 기록된다.

## 예시 자극 구조

* Go 자극: 과일
* No-Go 자극: 폭탄
* 또는 Go: 초록 불 / No-Go: 빨간 불
* 아동/청소년 버전과 성인 버전에서 그래픽만 바꾸고 규칙은 유지 가능

## 검사 모드 상세 사양

* 총 trial 수: 50
* Go 자극 비율: 75%
* No-Go 자극 비율: 25%
* 자극 표시 시간: 300ms
* 자극 간 간격(ISI): 800ms / 1000ms / 1200ms 랜덤
* 반응 허용 시간: 자극 시작 후 다음 자극 시작 전까지
* 연습 trial: 8
* No-Go는 연속 2회 이상 나오지 않도록 배치
* 검사 모드에서는 정답/오답 오버레이를 표시하지 않음

## 훈련 모드 상세 사양

* 초기 Go 비율은 높게 유지하여 반응 습관 형성
* 일정 점수 이상이면 No-Go 빈도와 자극 유사도 증가
* 연속 오류가 발생하면 금지 자극 사전 힌트 제공 가능

## 점수 계산 예시

* commission_rate = commission_errors / nogo_count
* fast_error_rate = 200ms 미만 반응 수 / 전체 반응 수
* mean_go_reaction_time를 함께 봐서 패턴을 해석한다
* commission_rate가 높고 mean_go_reaction_time이 400ms 미만이면 충동형 패턴으로 본다
* commission_rate가 높고 mean_go_reaction_time이 600ms 초과면 혼합형 패턴으로 본다

## 해석 원칙

* commission_errors가 많으면 억제 실패와 충동적 반응 경향 가능성이 높다.
* fast_error_rate가 높으면 성급하게 반응하는 경향을 의심할 수 있다.
* go_hit는 높지만 No-Go 실패가 많으면 반응 속도는 빠르지만 조절이 약한 패턴으로 해석한다.

## UI 요구사항

* 자극과 반응 버튼 영역을 분명하게 구분
* 검사 모드에서는 실시간 칭찬 요소 최소화
* 검사 모드에서는 성공/실패 오버레이를 표시하지 않음
* 자극이 사라진 뒤에도 입력 가능 시간이 남아 있음을 시각적으로 표시

## 저장 JSON 예시

```json
{
  "test_type": "go_nogo",
  "mode": "assessment",
  "go_count": 38,
  "nogo_count": 12,
  "go_hit_count": 34,
  "go_omission_count": 4,
  "commission_errors": 5,
  "mean_go_reaction_time": 481,
  "commission_rate": 0.42,
  "fast_error_rate": 0.08,
  "anticipatory_count": 1,
  "level_summary": {
    "pattern": "mixed"
  },
  "interpretation": "금지 자극에서의 억제 실패가 비교적 많아 충동 조절의 어려움을 시사합니다."
}
```

# 3. 과잉행동/자기조절 측정형 테스트: 균형 유지

## 목적

사용자의 활동성, 안절부절함, 미세한 움직임 조절, 안정 유지 능력을 측정한다.

## 핵심 측정 지표

* movement_amplitude: 움직임 크기 평균
* movement_variability: 움직임 흔들림 정도
* stable_hold_time: 목표 상태를 안정적으로 유지한 시간
* stable_duration_pct: 전체 시간 중 안정적으로 유지한 비율
* spike_count: 기준치를 넘는 큰 움직임 횟수
* total_movement: 전체 움직임 합
* raw_samples: 측정 샘플 배열

## 센서 사용 원칙

* 모바일 기기의 gyroscope와 accelerometer를 함께 사용한다.
* orientation 변화와 선형 가속도를 함께 저장할 수 있도록 설계한다.
* 센서 권한 획득 실패 시 테스트를 시작하지 않는다.
* 센서가 없으면 포인터 기반 fallback을 허용하되, 해석에서는 보조 참고 지표로 취급한다.

## 기본 규칙

* 사용자는 스마트폰을 손에 든 상태에서 화면 속 캐릭터 또는 공을 일정 영역 안에 유지해야 한다.
* 기기를 급하게 흔들거나 기울이면 캐릭터가 영역 밖으로 벗어난다.
* 목표는 빠른 반응이 아니라 안정 유지다.

## 검사 모드 상세 사양

* 총 길이: 30초
* 난이도: 고정
* 목표 영역 크기: 중간
* 시작 전 3초 카운트다운
* 센서 기준점 보정을 위한 초기 2초 정렬 단계 포함

## 훈련 모드 상세 사양

* 연속 성공 시 목표 영역 점진적 축소
* 유지 시간 증가
* 호흡 리듬 피드백 또는 배경 애니메이션 추가 가능
* 실패 시 목표 영역 잠시 확대

## 점수 계산 예시

* stability_score = stable_hold_time / total_duration
* movement_control_score는 movement_amplitude와 movement_variability가 낮을수록 높게 계산
* hyperactivity_proxy_score는 large_motion_count와 drift_distance가 높을수록 증가

## 해석 원칙

* movement_amplitude와 large_motion_count가 높으면 활동성 또는 자기조절 어려움 가능성을 시사한다.
* correction_count가 지나치게 많으면 안정 유지보다 과잉 보정 패턴을 의심할 수 있다.
* 이 테스트는 과잉행동을 단독으로 확정하는 도구가 아니라 활동성/자기조절 보조지표로 사용한다.

## UI 요구사항

* 화면 중앙 목표 영역 명확히 표시
* 현재 균형 유지 상태를 시각적으로 바로 보여줌
* 검사 모드에서는 점수 실시간 노출 금지
* 훈련 모드에서는 성공 시 부드러운 보상 애니메이션 허용

## 저장 JSON 예시

```json
{
  "test_type": "balance_hold",
  "mode": "assessment",
  "movement_amplitude": 0.84,
  "movement_variability": 0.31,
  "stable_hold_time": 18.4,
  "large_motion_count": 7,
  "drift_distance": 12.6,
  "correction_count": 14,
  "interpretation": "움직임 크기와 흔들림 빈도가 높아 활동성 및 자기조절의 어려움 가능성을 시사합니다."
}
```

# 통합 사용 원칙

## 세 테스트 결과의 역할

* 신호 찾기: 부주의 보조지표
* 멈춤 버튼: 충동성 보조지표
* 균형 유지: 활동성/자기조절 보조지표

## 통합 해석 원칙

* 최종 유형 분류는 설문 및 DSM-5 확인 결과를 기본 축으로 한다.
* 반응성 테스트는 유형 분류를 보강하는 객관 지표로 사용한다.
* 결과 화면에서는 “확정” 표현 대신 “경향”, “가능성”, “시사” 표현을 사용한다.

## 통합 JSON 예시

```json
{
  "reactivity_test_result": {
    "inattention_signal": {
      "score": 72,
      "source_test": "signal_detection"
    },
    "impulsivity_signal": {
      "score": 64,
      "source_test": "go_nogo"
    },
    "activity_signal": {
      "score": 58,
      "source_test": "balance_hold"
    },
    "summary": "부주의 관련 신호가 상대적으로 높고, 충동성은 중간 수준이며, 활동성 신호는 보조적으로 관찰됩니다."
  }
}
```

# 개발 우선순위 제안

1. 검사 모드 3종 구현
2. 공통 결과 JSON 구조 확정
3. 훈련 모드 추가
4. 결과 해석 문장 자동 생성 연결
5. 게이미피케이션 요소 추가

# 구현 시 주의사항

* 테스트 점수는 진단 확정 근거로 단독 사용하지 않는다.
* 검사 모드와 훈련 모드를 섞지 않는다.
* 접근성을 위해 색상 외 모양, 텍스트, 진동 등 대체 신호를 제공한다.
* 모바일 웹에서 센서 권한 허용/거부 흐름을 명확히 처리한다.
* 디버깅을 위해 raw event log 저장 옵션을 별도로 둔다.

# Codex 전달용 핵심 요약
