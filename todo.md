현재 구현과 문서 갱신 이후 남은 개선 과제를 정리합니다.

## 1. 명칭과 법적 안전성

- 사용자 화면에서는 `자가보고 선별`, `증상 기준 체크`로 바꿨습니다.
- 내부 키와 일부 폴더명은 호환성을 위해 `asrs`, `dsm5`를 유지합니다.
- 추후 외부 공개 전에는 문항 저작권/사용 허가 범위를 별도 검토해야 합니다.

## 2. 자가보고 선별

- 문항별로 "이 문항이 보는 영역" 라벨을 추가할지 검토합니다.
- 사용자군별 예시(직장인/학생/가정 상황)를 분기할지 결정합니다.
- quick analysis를 현재처럼 짧게 유지할지, plan과 연결되는 중간 제안을 추가할지 검토합니다.

## 3. 증상 기준 체크

- 시작 전 안내 카드 추가 여부를 검토합니다.
- 현재는 성인 흐름으로 정리되어 있으므로, 아동/청소년용 문구는 별도 버전으로 분리해야 합니다.
- 추가 확인 문항을 리포트의 일상 부담 계산에 얼마나 반영할지 기준을 더 세밀하게 조정할 수 있습니다.

## 4. 반응성 테스트 1: 신호 찾기

- 현재 자산 기준과 문서 기준은 정리됐지만, 목표/비목표 자극의 접근성 검토가 필요합니다.
- 색상만이 아니라 모양 차이로 충분히 구분되는지 확인해야 합니다.
- validity가 낮을 때 재시행 버튼을 제공할지 결정해야 합니다.

## 5. 반응성 테스트 2: 멈춤 버튼

- 현재 자산은 `o.gif`, `x.gif`이며 1024x1024, 25프레임, 0.50초, loop count 1 기준입니다.
- iPhone/iPad 터치 보정값 380ms가 실제 기기에서 적절한지 추가 테스트가 필요합니다.
- No-Go가 Go 2회 이상 뒤에 나오도록 완화했지만, 사용자 체감 난이도와 false alarm 비율을 계속 확인해야 합니다.
- success 기준 220ms 이후가 너무 관대하거나 엄격한지 실제 데이터 기반으로 재조정할 수 있습니다.

## 6. 반응성 테스트 3: 균형 유지

- 센서 기반, 롱터치, 포인터 fallback 간 점수 해석을 분리할 필요가 있습니다.
- 롱터치 방식은 과잉행동 직접 측정이 아니라 자기조절 보조 지표라는 설명을 계속 유지해야 합니다.
- 센서 입력에서 바닥에 내려놓은 상태나 지나치게 낮은 변동성 패턴을 감지하는 validity 규칙을 추가할 수 있습니다.

## 7. 결과 리포트

- 현재 리포트는 deterministic 계산을 우선합니다.
- 실제 사용자 데이터가 쌓이면 지표 임계값을 조정해야 합니다.
- 주관-객관 불일치가 있을 때 사용자에게 더 쉽게 설명하는 문구 개선이 필요합니다.

## 8. 실행 계획

- 현재 `buildPlanForMetrics()`가 유형별 fallback plan을 제공합니다.
- 사용자의 직업/생활 패턴을 입력받아 plan을 더 구체화할지 검토합니다.
- Gemini가 생성한 plan 문장이 의료 처방처럼 보이지 않는지 안전성 검토가 필요합니다.

## 9. 운영

- 관리자 버튼은 첫 화면에서 제거됐습니다.
- 향후 관리자 기능은 별도 URL 또는 인증 흐름으로 분리하는 것이 좋습니다.
- Cloudflare/Synology 배포 환경에서 HTTPS와 포트 프록시 설정 문서를 별도 운영 문서로 정리할 수 있습니다.
