# 질문과 해석

이 문서는 저장되는 JSON 값과 화면에서 보이는 해석 규칙을 정리합니다.

## 저장 원칙

- 저장 파일은 사용자 ID별 JSON입니다.
- 문항 전문과 raw 시행 기록은 필요한 범위에서 저장합니다.
- 리포트는 저장된 수치와 서버 계산값을 기준으로 생성합니다.
- JSON에 없는 값은 추정해서 리포트에 쓰지 않습니다.
- 사용자 화면에서는 `자가보고 선별`, `증상 기준 체크`라는 일반 명칭을 사용합니다.
- 내부 키는 호환성을 위해 `asrs`, `dsm5`를 유지합니다.

## 자가보고 선별

저장 예시:

```json
"asrs": {
  "responses": [1, 2, 3, 2, 4, 1],
  "attention_score": 8,
  "attention_max": 16,
  "hyperactivity_score": 5,
  "hyperactivity_max": 8,
  "total_score": 13,
  "total_max": 24,
  "positive_count": 4
}
```

값 설명:

- `responses`: 6개 문항의 원점수 배열
- `attention_score`: 1~4번 문항 합계
- `attention_max`: 1~4번 문항 최대 점수
- `hyperactivity_score`: 5~6번 문항 합계
- `hyperactivity_max`: 5~6번 문항 최대 점수
- `total_score`: 전체 합계
- `positive_count`: 기준 이상 응답 개수

해석 규칙:

- 1~3번은 2점 이상이면 유의미 응답
- 4~6번은 3점 이상이면 유의미 응답
- `positive_count >= 4`면 추가 평가를 고려할 만한 선별 신호로 봅니다.

## 증상 기준 체크

저장 예시:

```json
"dsm5": {
  "responses": [true, false, true],
  "inattention_true_count": 5,
  "hyperactivity_true_count": 4,
  "contextual_true_count": 4,
  "total_true_count": 9,
  "subtype": "판정 보류"
}
```

값 설명:

- `responses`: 전체 문항 Yes/No 배열
- `inattention_true_count`: 부주의 문항 Yes 개수
- `hyperactivity_true_count`: 과잉행동/충동성 문항 Yes 개수
- `contextual_true_count`: 추가 확인 문항 Yes 개수
- `total_true_count`: 주요 증상 문항 Yes 총합
- `subtype`: 현재 규칙으로 계산한 유형 문자열

해석 규칙:

- 부주의 6개 이상, 과잉행동/충동성 6개 미만: `부주의형 가능성`
- 과잉행동/충동성 6개 이상, 부주의 6개 미만: `과잉행동·충동형 가능성`
- 두 영역 모두 6개 이상: `복합형 가능성`
- 두 영역 모두 6개 미만: `무증상 범위`
- 응답 미완료: `판정 보류`

추가 확인 문항은 유형을 직접 정하지 않고 일상 부담과 맥락 해석을 보강합니다.

## 반응성 테스트

### 신호 찾기

핵심 저장값:

- `target_count`: 목표 자극 수
- `hit_count`: 목표에 올바르게 반응한 수
- `omission_errors`: 목표를 놓친 수
- `omission_rate`: 목표 놓침 비율
- `false_alarm_count`: 비목표 오반응
- `reaction_time_variability`: 반응시간 변동성
- `tau`: 느린 반응 꼬리
- `late_phase_drop`: 후반부 정확도 저하
- `anticipatory_count`: 100ms 미만 반응

해석:

- 점수보다 `omission_rate`, `reaction_time_variability`, `tau`, `late_phase_drop`를 우선 봅니다.
- 예측 반응이 많으면 validity가 낮아질 수 있습니다.

### 멈춤 버튼

핵심 저장값:

- `go_count`: 눌러야 하는 시행 수
- `nogo_count`: 누르면 안 되는 시행 수
- `go_hit_count`: Go 성공 수
- `go_omission_count`: Go 놓침 수
- `commission_errors`: No-Go에서 누른 수
- `commission_rate`: 잘못된 반응 비율
- `fast_error_rate`: 성급 반응 비율
- `premature_response_count`: Go 성공 기준보다 빠른 반응
- `mean_go_reaction_time`: Go 평균 반응시간

해석:

- `commission_rate`가 높으면 멈춤 조절 어려움을 봅니다.
- `fast_error_rate`와 `premature_response_count`가 높으면 성급 반응을 봅니다.
- `mean_go_reaction_time`과 함께 봐서 충동형/혼합형/균형형 패턴을 설명합니다.

### 균형 유지

핵심 저장값:

- `stable_hold_time`: 안정 유지 시간
- `stable_duration_pct`: 안정 유지 비율
- `spike_count`: 큰 흔들림 횟수
- `total_movement`: 전체 움직임 크기
- `input_source`: `sensor`, `long_touch`, `pointer`

해석:

- 이 테스트는 활동성/자기조절 보조 지표입니다.
- 센서 기반이 아니면 해석 강도를 낮게 둡니다.
- 과잉행동을 직접 확정하는 지표로 사용하지 않습니다.

## 통합 리포트

리포트는 다음 값을 종합합니다.

- 자가보고 부주의/충동성 점수
- 증상 기준 체크 Yes 개수
- 신호 찾기 부주의 지표
- 멈춤 버튼 충동성 지표
- 균형 유지 활동성 지표

리포트 구조:

- `hero`: 전체 요약
- `crossCheck`: 주관적 보고와 객관적 반응 비교
- `profile`: 부주의/충동성 세부 설명
- `dailyImpact`: 일상 부담
- `bridge`: plan 이동 CTA

## 실행 계획

계획은 `plan.suggestions` 3개와 `plan.chat`으로 저장됩니다.

분기:

- 부주의 경향: 과제 분할, 시각적 단서, 환경 통제
- 충동성 경향: 심호흡, 지연 행동, 자기 점검
- 복합형 경향: 두 전략 혼합
- 안정 범위: 유지 루틴

계획은 의료 처방이 아니라 일상 코칭 문장으로 작성됩니다.
